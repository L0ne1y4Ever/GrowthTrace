# GrowthTrace 联调与启动说明

## 1. 数据准备

创建数据库：

```sql
CREATE DATABASE growthtrace DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

导入结构与种子数据：

```powershell
Get-Content sql\schema.sql | mysql -h127.0.0.1 -P3306 -uroot -p growthtrace
Get-Content sql\seed.sql | mysql -h127.0.0.1 -P3306 -uroot -p growthtrace
```

## 2. 后端启动

推荐方式：在 `backend/src/main/resources/application-dev.local.yml` 放本地专用配置。

示例：

```yml
spring:
  datasource:
    username: root
    password: 你的密码

growthtrace:
  security:
    jwt:
      secret: 长度至少 32 位的随机字符串
  ai:
    base-url: https://api.openai.com/v1
    api-key: 你的密钥
    model: gpt-4o-mini
    connect-timeout-seconds: 10
    timeout-seconds: 90
```

也可以使用环境变量。PowerShell 示例：

```powershell
$env:DB_USER='root'
$env:DB_PASSWORD='你的密码'
$env:JWT_SECRET='长度至少 32 位的随机字符串'
$env:AI_BASE_URL='https://api.openai.com/v1'
$env:AI_API_KEY='你的密钥'
$env:AI_MODEL='gpt-4o-mini'
$env:GROWTHTRACE_AI_CONNECT_TIMEOUT_SECONDS='10'
$env:GROWTHTRACE_AI_TIMEOUT_SECONDS='90'

cd backend
mvn spring-boot:run
```

默认地址：

- API：`http://localhost:8080/api`

## 3. 前端启动

## 6. 常见问题

### 6.1 后端启动报数据库连接失败

优先检查：

- MySQL 是否运行在 `127.0.0.1:3306`
- `DB_USER / DB_PASSWORD` 是否正确
- `growthtrace` 数据库是否已导入

### 6.2 AI 接口调用失败

优先检查：

- `AI_API_KEY` 是否有效
- `AI_BASE_URL` 是否为 OpenAI-compatible 地址
- `AI_BASE_URL` 是否指向 `/chat/completions` 的上一级
- `AI_MODEL` 是否是当前平台真实可用的模型标识
- 当前网络是否能访问对应模型服务

常见 OpenAI-compatible 地址示例：

- OpenAI：`https://api.openai.com/v1`
- SiliconFlow：`https://api.siliconflow.cn/v1`

### 6.3 AI 排障顺序

遇到“没调用记录 / 超时 / 返回格式异常”时，按下面顺序查：

1. 浏览器 Network 是否发出了 `POST /api/profile/onboarding/extract`、`/journal/{id}/extract`、`/diagnosis/trigger`
2. 后端日志是否出现 `AI request start`
3. `AI config summary` 里的 `baseUrl`、`model`、`apiKeyConfigured` 是否正确
4. 后端日志里是：
   - `AI request timeout`
   - `AI request failed: status=...`
   - `AI 响应缺少 choices[0].message.content`
5. 如果是 404，优先检查 `base-url` 是否少了 `/v1`
6. 如果是 401/403，优先检查 `api-key`
7. 如果是 parse error，优先检查模型是否真的兼容 OpenAI Chat Completions 响应

### 6.4 前端登录后接口 401

优先检查：

- 后端是否成功启动
- 浏览器本地 token 是否过期
- `JWT_SECRET` 是否在重启后变更，导致旧 token 失效
