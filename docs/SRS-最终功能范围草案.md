# GrowthTrace SRS 最终功能范围草案

## 1. 定位

`GrowthTrace` 是一个面向计算机专业学生的成长跟踪与阶段诊断系统。

系统目标不是做单点工具，而是把学生成长过程中的关键信息串成一个可持续闭环：

- 建立成长画像
- 设定阶段目标
- 拆解为执行任务
- 记录成长随记
- 通过 AI 做辅助抽取与阶段总结
- 形成诊断与纠偏动作
- 回流到下一轮执行

## 2. 建议锁定的主线

建议在 `SRS` 中明确本系统的核心业务主线为：

`画像 -> 目标 -> 要求 -> 任务 -> 随记 -> AI 抽取确认 -> 阶段诊断 -> 纠偏动作 -> 再执行`

这个主线决定后续所有模块设计都要围绕“闭环”展开，而不是各模块独立存在。

## 3. 核心功能范围

### 3.1 认证模块

功能范围：

- 用户注册
- 用户登录
- 获取当前登录用户信息
- 基于 JWT 的身份认证

目标：

- 为后续成长数据建立用户隔离和身份基础

### 3.2 成长画像模块

功能范围：

- 用户使用自然语言进行首次建档
- AI 抽取画像草稿
- 用户确认后正式入档
- 维护技能、经历、方向、当前状态等信息
- 更新画像完整度

目标：

- 形成目标设定和阶段诊断的基础输入

### 3.3 目标管理模块

功能范围：

- 创建和维护成长目标
- 为目标定义 requirement / milestone
- 标记主目标
- 更新 requirement 完成状态

目标：

- 让成长过程具备阶段性目标与可衡量标准

### 3.4 执行任务模块

功能范围：

- 创建、修改、删除执行任务
- 设置任务优先级、截止日期、状态
- 任务打卡与进度记录
- 周进度统计

目标：

- 把目标要求转成可执行的具体行动

### 3.5 成长随记模块

功能范围：

- 编写成长随记
- 记录情绪、标签、内容摘要
- 触发 AI 抽取成长事件草稿
- 用户确认抽取结果

目标：

- 保留成长过程中的原始行为证据和上下文

### 3.6 阶段诊断模块

功能范围：

- 触发阶段诊断
- 计算规则指标
- 调用 AI 生成阶段总结、建议和纠偏方向
- 查看诊断历史
- 记录复盘内容

目标：

- 把离散成长数据转成阶段性结论和纠偏建议

### 3.7 Dashboard 总览模块

功能范围：

- 展示画像完整度
- 展示当前主目标
- 展示本周任务进展
- 展示随记与成长趋势
- 展示最近一次诊断结果
- 展示下一步建议动作

目标：

- 成为系统的统一行动入口，而不是纯展示页面

## 4. 建议纳入 SRS 的“强联动”功能

这部分是当前系统最值得补强、也最能让整体不显松散的地方。建议作为本系统的重要功能特性写入 `SRS`。

### 4.1 目标要求与执行任务联动

建议写入：

- 用户可基于目标 requirement 创建执行任务
- 任务可关联到具体目标与 requirement
- 系统可通过任务完成情况辅助判断目标推进状态

意义：

- 让目标不只是描述，而能落实到执行层

### 4.2 随记与目标/任务联动

建议写入：

- 随记可关联当前目标、requirement 或任务
- AI 抽取时优先识别该随记影响了哪些 requirement、技能或阻塞项
- 确认后的抽取结果可回流到画像与目标推进状态

意义：

- 让随记成为成长证据，而不是孤立日志

### 4.3 诊断结果与执行动作联动

建议写入：

- 诊断结果除阶段总结外，还应给出下一步行动建议
- 用户可依据诊断建议调整任务和目标要求
- 后续版本可支持从建议生成任务

意义：

- 让诊断从“报告”变成“行动驱动器”

### 4.4 Dashboard 与闭环联动

建议写入：

- Dashboard 集中展示待确认抽取草稿、本周任务、最近诊断和下一步建议动作
- 用户可从 Dashboard 直接跳转到下一步最需要处理的事项

意义：

- 把系统首页从信息汇总页提升为流程入口页

## 5. 建议作为“增强需求”写入 SRS 的内容

这些内容可以作为本期增强功能，不建议写成完全独立的新模块。

- 诊断历史对比
- 成长趋势可视化增强
- 阶段诊断结果导出
- 周报或成长摘要导出
- 更明确的任务优先级与提醒策略

## 6. 不建议纳入本期核心范围的内容

为避免课设范围失控，以下内容建议在 `SRS` 中列为“非本期范围”或“后续扩展”：

- 管理员后台
- 多角色复杂权限
- 微服务拆分
- 独立移动端重做
- 即时通知系统
- 重型算法推荐模块

## 7. 非功能需求建议

### 7.1 架构

- 采用 `B/S` 架构
- 前后端分离实现
- 前端提供用户交互与展示
- 后端负责业务逻辑、权限、数据持久化和 AI 调用适配

### 7.2 性能

- 普通查询类接口应具备较快响应速度
- AI 相关接口允许更长等待时间，但需提供明确提示
- 系统应支持课程演示场景下的稳定运行

### 7.3 可用性

- 用户界面应清晰展示当前状态
- AI 失败时应给出可理解的错误提示
- 核心闭环页面应可直接跳转到下一步操作

### 7.4 安全性

- 不同用户的数据需隔离
- 认证令牌需安全保存与校验
- AI 密钥、数据库密码等敏感配置不得写入公开仓库

### 7.5 可维护性

- 按模块分层组织代码
- 提供明确文档、接口说明和联调说明
- 支持后续扩展功能迭代

## 8. 建议的 SRS 定稿策略

建议把 `SRS` 定在以下口径：

- 基础模块全部纳入
- 强联动能力写进正式需求
- 增强需求只保留 2 到 4 个
- 明确列出非本期范围

这样做的好处是：

- 范围稳定
- 闭环更强
- 文档更有说服力
- 后续设计、实现、测试、答辩更容易一致
