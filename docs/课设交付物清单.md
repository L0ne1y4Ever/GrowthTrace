# GrowthTrace 课设交付物清单

> 本文档用于项目组内部推进和材料补齐，不直接作为正式课程报告正文。正式提交文档应统一采用“需求先行、后续开发拟开展”的写法。

## 目标

把当前 `GrowthTrace` 从“项目可运行”推进到“符合软件工程课程设计 II 提交要求”的状态。

当前判断：

- 系统规模基本达标
- 模块数量达标
- 数据表数量达标
- `B/S` 架构在老师最新口径下可接受
- 当前主要缺口在课程化文档、团队过程材料、最终报告组织

## 一、已满足项

### 1. 系统规模

- 已有 7 个核心模块：
  - `auth`
  - `profile`
  - `journal`
  - `target`
  - `execution`
  - `diagnosis`
  - `dashboard`
- 已有 11 张核心数据表
- 已有前后端分离完整工程

### 2. 功能闭环

- 已形成主链路：
  - 建档
  - 目标设定
  - 随记记录
  - AI 抽取确认
  - 任务执行
  - 阶段诊断
  - Dashboard 总览

### 3. 验收基础

- 数据库已可初始化
- 登录注册已可用
- 关键接口可联调
- 前端可构建
- 后端测试可运行
- 已有演示脚本和验收清单

## 二、部分满足项

### 1. 创新性

- 当前创新点：
  - 面向计算机专业学生成长跟踪，而不是通用管理系统
  - 引入 AI 做建档抽取、随记抽取、阶段诊断
  - 强调“AI 抽取 + 用户确认”的安全闭环
- 后续需要在答辩材料里把创新点讲得更明确：
  - 应用场景创新
  - 业务流程创新
  - 与普通学生信息管理系统的差异

### 2. 软件工程过程材料

- 当前已有零散文档，但还未形成完整课程提交包
- 需要从“开发文档”整理成“课设文档”

## 三、当前缺口

### 1. 需求规格说明书（SRS）

- 需要基于模板完成正式版需求文档
- 原模板不能直接改
- 已复制副本：
  - [GrowthTrace-软件需求规格说明书.doc](/D:/GrowthTrace/resource/GrowthTrace-%E8%BD%AF%E4%BB%B6%E9%9C%80%E6%B1%82%E8%A7%84%E6%A0%BC%E8%AF%B4%E6%98%8E%E4%B9%A6.doc)
- 原模板保留：
  - [2022软件需求规格说明书（SRS）模板.doc](/D:/GrowthTrace/resource/2022%E8%BD%AF%E4%BB%B6%E9%9C%80%E6%B1%82%E8%A7%84%E6%A0%BC%E8%AF%B4%E6%98%8E%E4%B9%A6%EF%BC%88SRS%EF%BC%89%E6%A8%A1%E6%9D%BF.doc)

### 2. 系统设计说明

- 需要补以下内容：
  - 系统总体架构图
  - 模块划分图
  - 业务流程图
  - 数据库 ER 图
  - 关键时序图
  - 主要接口设计说明

### 3. 测试与验收材料

- 现有内容偏工程内部文档
- 需要整理成课程报告可直接引用的章节：
  - 测试环境
  - 功能测试
  - 接口测试
  - AI 链路测试
  - 验收结论

### 4. 团队过程材料

- 必须补足不少于 3 次团队活动记录
- 每次活动至少要有：
  - 时间
  - 地点
  - 参与人
  - 讨论内容
  - 分工结论
  - 现场照片

### 5. 团队与个人分工说明

- 需要有一份单独材料写清：
  - 组长职责
  - 每位成员负责的模块
  - 每位成员的实际工作量
  - 报告中每个人编写的部分

### 6. 最终课程报告

- 需要整合成不少于 30 页的正式提交材料
- 建议结构：
  - 1. 项目背景与目标
  - 2. 可行性与需求分析
  - 3. 总体设计
  - 4. 数据库设计
  - 5. 详细设计与核心实现
  - 6. 测试与验收
  - 7. 创新点总结
  - 8. 团队协作与分工
  - 9. 个人总结

## 四、建议优先级

### P0：先补正式提交必需件

1. 完成 SRS
2. 完成系统设计说明
3. 完成团队分工说明
4. 整理 3 次团队活动记录

### P1：再补答辩与报告内容

1. 汇总测试与验收结果
2. 提炼创新点
3. 形成课程报告正文
4. 形成答辩 PPT

### P2：最后做展示增强

1. 补系统截图
2. 补架构图、ER 图、流程图
3. 补演示视频或录屏素材

## 五、建议推进顺序

### 阶段 1：锁定文档骨架

- 完成 SRS 目录和主要内容
- 明确最终报告目录
- 明确团队分工表

### 阶段 2：补课程化设计材料

- 架构设计
- 数据库设计
- 功能设计
- 测试设计

### 阶段 3：整理最终提交包

- 统一格式
- 插入图片与图表
- 补团队活动记录
- 补个人总结

## 六、下一步建议

当前最值得先做的是：

1. 先读取并整理 `SRS` 模板结构
2. 基于当前系统内容填写 `GrowthTrace-软件需求规格说明书.doc`
3. 再根据 `SRS` 反推设计说明书和最终课程报告

这条顺序最稳，因为 `SRS` 会决定后面大部分课程文档的边界和口径。
